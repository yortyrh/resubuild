## When to use

- The operator (or an LLM agent) wants to deploy the resubuild system to a managed Railway project connected to a non-self-hosted (cloud) Supabase project.
- The operator has accepted the limitations of the release-1 Railway target (no persistent Puppeteer cache volume on the free tier, no automated custom-domain TLS, no preview environments, public-URL handshake) and wants the standard two-service topology.
- The operator has a `prod-secrets.json` manifest on disk (or is willing to write one) and is ready to walk the seven steps in order.

## When NOT to use

- For local development — use `pnpm setup:env` (which runs `bash scripts/setup-local-env.sh`) instead.
- For self-hosted docker compose production — use the `setup-prod-env` SKILL and the existing `docker-compose.prod.yml` target. The Railway target is a parallel option, not a replacement.
- For TLS-terminated, multi-service production with persistent volumes, custom domains, or preview environments — the release-1 Railway target intentionally does not cover these (see **Known limitations** below). Use a different target or wait for a follow-on release-1 change.
- If the operator wants the SKILL to invoke `railway up` itself — it will not. The wrapper script (`pnpm deploy:railway`) only runs the preflight check; the operator runs the two `railway up --service <name>` commands interactively so they can watch the build log.

## Inputs

The SKILL walks the operator through the same env-var surface as the docker compose target. Inputs are gathered by the existing `setup-prod-env` SKILL and written to `prod-secrets.json` (gitignored) — this SKILL does not re-prompt for them. Required inputs:

1. **Supabase project URL** — from the Supabase dashboard → Project Settings → API → "Project URL"
2. **Supabase anon key** — Project Settings → API → "anon / public" key
3. **Supabase service role key** — Project Settings → API → "service_role" key (treat as server-only secret)
4. **Two Storage bucket names** — `media` and `mcp-exports` (must exist in the Supabase project)
5. **AI_AGENT_ENCRYPTION_KEY** — AES-256 key, auto-generated by the script if absent
6. **DEPLOY_TARGET** — `railway` (so the generator bakes the production custom-domain defaults)

The four public-URL keys (`CORS_ORIGIN`, `APP_URL`, `PUBLIC_API_URL`, `NEXT_PUBLIC_API_URL`) are written by the generator with the **production custom domains** when `--target railway` is used:

- `CORS_ORIGIN` / `APP_URL` → `https://app.resubuild.dev` (the web app)
- `PUBLIC_API_URL` / `NEXT_PUBLIC_API_URL` → `https://api.resubuild.dev` (the API)

The operator MAY override any of these in the manifest, but the defaults are already the real production domains, so a freshly generated `.env.prod` is immediately deployable as long as the operator has attached the `app.resubuild.dev` and `api.resubuild.dev` custom domains to the corresponding Railway services (Step 4c).

## Workflow

The seven steps below MUST be run in order. The SKILL never echoes secret values back to the user — every secret flows through the on-disk `prod-secrets.json` manifest.

### Step 1 — Generate `.env.prod` with the Railway target

If a `prod-secrets.json` manifest does not exist, write one (see the `setup-prod-env` SKILL for the JSON shape). Make sure the manifest sets `"DEPLOY_TARGET": "railway"` (or pass `--target railway` on the CLI below).

Dry-run first for review:

```bash
pnpm setup:env:prod --target railway --from prod-secrets.json --dry-run
```

After the operator confirms the output, write the file:

```bash
pnpm setup:env:prod --target railway --from prod-secrets.json
```

The four public-URL keys are written with the production custom domains (`app.resubuild.dev` for the web app, `api.resubuild.dev` for the API). If the operator has attached those custom domains to the corresponding Railway services (Step 4c), the file is immediately deployable — no find-and-replace step.

### Step 2 — Install the Railway CLI

```bash
npm install -g @railway/cli
# or
brew install railway
```

### Step 3 — Create the Railway project

Log in (interactive — opens a browser):

```bash
railway login
```

Create a fresh project from the dashboard (recommended over `railway init` for a "one project, two services from the same repo" topology):

1. Open https://railway.com/new
2. Name the project (e.g. `resubuild`)
3. Skip the empty-service template — we will add two services from the existing repo

### Step 4 — Link the repo and add the two services

Link the project from the repo root:

```bash
railway link
```

**4a. Add the api service** (service root at `apps/api`, config at `/apps/api/railway.json`):

1. In the Railway dashboard, click **+ New** → **GitHub Repo** (or **Empty Service** if linking manually).
2. Select this repo and pick the `apps/api` directory as the service root.
3. In the service's **Settings** tab, scroll to **Config as Code** and set the config file path to `/apps/api/railway.json`.
4. Railway reads the file and pins the build (`apps/api/Dockerfile`) and the start command (`node apps/api/dist/main`).

**4b. Add the web service** (service root at `apps/web`, config at `/apps/web/railway.json`): the start command becomes `pnpm --filter @resubuild/web start`.

**4c. Attach the custom domains** (one-time per service). The four public-URL keys in `.env.prod` default to the production custom domains (`app.resubuild.dev`, `api.resubuild.dev`); they will not work until the matching domain is attached to the service:

1. In the api service's **Settings** tab → **Networking** → **Custom Domain**, attach `api.resubuild.dev`.
2. Repeat for the web service with `app.resubuild.dev`.
3. Railway provisions a Let's Encrypt certificate automatically — no manual TLS step.

**4d. Enable App Sleeping (scale to zero).** The release-1 Railway target is designed to scale from zero when idle. This is a per-service dashboard toggle (not in `railway.json`):

1. In each service's **Settings** tab, scroll to **App Sleeping**.
2. Toggle **Enable App Sleeping** on.
3. Trigger a new deployment for the change to take effect.

When App Sleeping is on, Railway puts the service to sleep after 10 minutes of outbound inactivity and wakes it on the next inbound request. Expect a 30-60s cold start on the api service (Puppeteer Chromium re-download) and a sub-second cold start on the web service.

### Step 5 — Paste per-service Variables

For each service, open the **Variables** tab and paste the corresponding subset of `.env.prod`. Both services share the Supabase / Storage / AI keys. The public-URL keys are split:

**api service Variables:**

```
SUPABASE_URL
SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
SUPABASE_JWT_SECRET          (optional)
MEDIA_BUCKET
MCP_EXPORT_BUCKET
PORT                        (Railway injects this; leave alone)
CORS_ORIGIN                 = https://app.resubuild.dev
APP_URL                     = https://app.resubuild.dev
PUBLIC_API_URL              = https://api.resubuild.dev
AI_AGENT_ENCRYPTION_KEY
MCP_KEY_PEPPER              (optional; defaults to SUPABASE_SERVICE_ROLE_KEY)
MCP_SERVER_ENABLED          (default true)
MCP_EXPORT_TTL_SECONDS      (default 3600)
MCP_EXPORT_MAX_BYTES        (default 10485760)
PDF_IMPORT_MAX_BYTES        (default 5 MiB)
PDF_IMPORT_ENABLED          (default true)
```

**web service Variables:**

```
NEXT_PUBLIC_API_URL         = https://api.resubuild.dev
```

(Only the public API URL needs to be in the browser bundle. All other keys are api-side.)

The four public-URL values are the production custom domains attached in Step 4c. They match the defaults the generator wrote, so pasting the same value is correct. Save the variables; Railway triggers a rebuild.

### Step 6 — Run the preflight and deploy

Confirm `.env.prod` is well-formed (the same validator the script uses):

```bash
pnpm deploy:railway
```

The script exits zero and prints:

```
railway up --service api
railway up --service web
```

Run each `railway up` command interactively (one at a time) and watch the build log. The api service's first build is the slow one (~3-5 min, downloads Chromium). The web service builds faster.

No find-and-replace step is needed — the `.env.prod` written in Step 1 already has the production custom domains, which match the Railway Variables pasted in Step 5.

### Step 7 — Verify

```bash
curl -f https://api.resubuild.dev/_health
curl -f https://app.resubuild.dev/
```

The first should return `{"status":"ok",...}`. The second should return the HTML shell of the resubuild web app. Sign in with the developer account to confirm the full stack is reachable.

## Known limitations

These are **scope statements**, not TODOs. The release-1 Railway target is intentionally minimal.

- **No persistent Puppeteer cache volume** on the Railway free tier. The api service re-downloads Chromium (~181 MiB) on every cold start from App Sleeping (and on every fresh deploy). Operators on a paid Railway plan can attach a volume at the path `/root/.cache/puppeteer` in the api service's **Settings** → **Volumes** tab to skip the re-download.
- **No preview / ephemeral environments per pull request.** The Railway target deploys the linked branch only (typically `main`). Preview environments are a follow-on release-1 change.
- **Public-URL handshake.** The two services communicate over the public internet (or, on paid plans, over a Railway private network), not over a docker compose bridge network. CORS must be configured for the public origin (`CORS_ORIGIN` in the api service). This is by design for a managed-platform target.
- **App Sleeping is a per-service dashboard toggle, not a `railway.json` field.** The deploy artifacts document the toggle in Step 4d; it is not enforced by config-as-code. Operators MUST enable it manually per service or the service will not scale to zero.

## Guardrails

### NEVER echo secret values back to the user

Do not print, repeat, or summarize secret values (especially `SUPABASE_SERVICE_ROLE_KEY`, `AI_AGENT_ENCRYPTION_KEY`, or any key that starts with `eyJ`). If the user asks what a secret value is, direct them to the manifest file on disk.

### Use the manifest file as the only safe place for secret material

The SKILL must write secrets to `prod-secrets.json` on disk (which is gitignored). The script reads from the manifest, not from the chat. This prevents secret values from appearing in chat transcripts.

### If the user pastes a secret into chat

1. Do NOT repeat or echo it back.
2. Tell the user to revoke the key in the Supabase dashboard and generate a new one.
3. Advise them to use the manifest file approach instead of pasting secrets into chat.

### Always run `--dry-run` first

The SKILL MUST invoke the dry-run first and wait for operator confirmation before running without `--dry-run`. Operators can otherwise miss a wrong public-URL placeholder template.

### The SKILL never invokes `railway up` directly

The deploy commands are interactive and service-scoped. The operator runs them manually so they can watch the build log and respond to platform prompts. The preflight wrapper (`pnpm deploy:railway`) is a guard, not a deploy driver.

### The `.railway/` directory is machine-local

`railway link` writes `.railway/config.json` and friends to the repo root. This directory is gitignored. The operator MUST NOT commit its contents.
