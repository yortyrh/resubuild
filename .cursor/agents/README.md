# OpenSpec Parallel Subagents

Configuration for running multiple OpenSpec changes in parallel using **Cursor subagents** + **git worktrees**.

Reference: <https://cursor.com/docs/subagents>

---

## Why

OpenSpec changes are independent units of work. With 10 active changes and only one working tree, contributors have to serialize implementation. This setup lets the agent:

1. **Analyze** all (or selected) changes for file-touch conflicts
2. **Partition** them into batches that can safely run in parallel
3. **Spawn** one implementer subagent per change, each in its own git worktree

Each subagent has its own context window (no token bleed between changes) and its own filesystem checkout (no merge collisions during implementation).

---

## Files in this folder

| File                         | Purpose                                                                                                                     |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `opsx-parallel-planner.md`   | Read-only subagent. Builds a conflict graph from change artifacts and emits `state/parallel-plan.json`.                     |
| `opsx-change-implementer.md` | Background, write-enabled subagent. Implements one change end-to-end inside its assigned worktree and commits once locally. |
| `state/parallel-plan.json`   | Generated by the planner. Consumed by the launcher. Gitignored.                                                             |

Related slash commands live in `../commands/`:

| Command                  | Use                                                                                    |
| ------------------------ | -------------------------------------------------------------------------------------- |
| `/opsx-plan-parallel`    | Plan parallel work. Optionally accepts a change list or `--exclude` filter.            |
| `/opsx-launch-parallel`  | Execute the plan. Supports `--batch N`, `--changes a b c`, `--dry-run`.                |
| `/opsx-parallel-status`  | Inspect every active worktree (tasks, commits, dirty state). Read-only.                |
| `/opsx-parallel-cleanup` | Remove worktrees safely. `--all-merged`, `--force`, `--delete-branch` flags available. |

## Standard workflow

```
/opsx-plan-parallel                  → planner subagent → state/parallel-plan.json
                                    ↓
                       (review the Markdown summary)
                                    ↓
/opsx-launch-parallel                → creates .worktrees/<change>/ per change,
                                       spawns N implementer subagents in parallel,
                                       each commits ONCE in its worktree
                                    ↓
/opsx-parallel-status                → see what landed, what's dirty, what paused
                                    ↓
                git checkout main && git merge --no-ff opsx/<change>    ← you, manually
                                    ↓
/opsx-parallel-cleanup --all-merged  → remove worktrees of merged branches
```

## Layout on disk

After `/opsx-launch-parallel` runs:

```
jsonresume-web/                          ← main worktree (you stay here)
├── .git/
├── .worktrees/                          ← gitignored
│   ├── sidebar-tab-navigation/          ← branch: opsx/sidebar-tab-navigation
│   │   ├── apps/ packages/ openspec/    ← full checkout
│   │   └── node_modules/                ← per-worktree
│   ├── language-select-filter/          ← branch: opsx/language-select-filter
│   └── markdown-view-rendering/         ← branch: opsx/markdown-view-rendering
├── apps/
└── ...
```

- **One worktree = one branch = one change.** Branches are named `opsx/<change-name>`.
- Worktrees live inside the repo at `.worktrees/` (gitignored) so they're easy to discover, inspect with `ls`, and clean up.
- Each worktree has its own `node_modules` — pnpm install runs once per worktree on first launch.

## Subagent design

### Planner (`opsx-parallel-planner`)

- **Readonly: true** — cannot create branches, worktrees, or modify code
- **Model: inherit** — uses the parent agent's model
- Reads `tasks.md`, `proposal.md`, `design.md` for each change
- Extracts touched file paths and classifies conflicts:
  - **Hard** (same file → must be in different batches)
  - **Soft** (`pnpm-lock.yaml`, parallel migrations → allowed but flagged)
- Outputs a deterministic batch partition + JSON plan + Markdown summary

### Implementer (`opsx-change-implementer`)

- **Readonly: false** — writes code, runs tests, commits
- **Is_background: true** — non-blocking, so the launcher can spawn many in one message
- **Model: inherit**
- Receives a JSON contract: `{ change, branch, worktreePath, absoluteWorktreePath }`
- Constraints:
  - Stays inside its worktree (never `cd` out)
  - Never pushes (no remote operations)
  - Never touches `main`
  - One commit per change (atomic, conventional commit format)
  - Disables git hooks for its commit (the launcher runs `pnpm verify` before merging anyway)
  - Reports back as JSON for the orchestrator to parse

## Conflict-handling strategy

The planner does its best with static analysis of `tasks.md` paths, but file-path conflicts can still slip through. The launcher mitigates this by:

1. **Running batches sequentially.** Members within a batch are parallel; batches themselves are serial.
2. **Not auto-merging.** All commits stay on `opsx/*` branches locally. You merge them manually in any order you choose — `git` will tell you about conflicts at merge time.
3. **Hard rule: no pushes.** Nothing leaves your machine until you decide.

If a merge later fails:

```bash
git checkout main
git merge --no-ff opsx/sidebar-tab-navigation     # easy one first
git merge --no-ff opsx/markdown-view-rendering    # may conflict on cv-sections.tsx
# resolve, commit, continue
```

## Operational notes

### Disk cost

Each worktree is a full checkout + its own `node_modules` + `.next` + `.turbo`. Expect ~500 MB per worktree on this monorepo. Clean up with `/opsx-parallel-cleanup --all-merged` once branches are merged.

### Token cost

Subagents consume tokens independently. Running 5 implementers in parallel costs roughly 5× the tokens of one. The launcher caps parallel implementers at **6 by default** — beyond that, batches are chunked with user confirmation.

### Resuming a paused implementer

If an implementer pauses (e.g. ambiguous task), the worktree stays put with progress checkpointed in `tasks.md`. Resume by spawning the same subagent again with the same input JSON — it will pick up from the next unchecked task. (Or use the worktree directly in your IDE and finish manually.)

### Built-in subagents still work

This config doesn't replace Cursor's built-in `explore`, `bash`, `browser` subagents. Those auto-trigger for context-heavy operations within both the main agent and the implementers.

---

## Don'ts (hard rules baked into the subagents)

- **Don't push.** Implementers and the launcher are forbidden from `git push` and `gh pr create`.
- **Don't merge.** Branches sit on `opsx/*` until you manually merge.
- **Don't modify another worktree.** Each implementer is fenced to its own `absoluteWorktreePath`.
- **Don't auto-resolve merge conflicts.** Surface them and stop.
- **Don't run `pnpm verify` from a worktree against the main repo.** Worktrees test themselves only.

## Quick recipes

**Plan everything, run batch 1 only, see status, clean up merged work:**

```
/opsx-plan-parallel
/opsx-launch-parallel --batch 1
# ... merge a couple of branches manually ...
/opsx-parallel-status
/opsx-parallel-cleanup --all-merged
```

**Plan just three specific changes:**

```
/opsx-plan-parallel sidebar-tab-navigation language-select-filter markdown-view-rendering
/opsx-launch-parallel
```

**Dry-run to see what would happen:**

```
/opsx-plan-parallel
/opsx-launch-parallel --dry-run
```

**Force-skip the planner's batching (you've manually verified no conflicts):**

```
/opsx-launch-parallel --changes change-a change-b change-c
```

## Customizing

- Edit the subagent markdown files in this folder. Frontmatter knobs: `model`, `readonly`, `is_background`. See the [Cursor subagents docs](https://cursor.com/docs/subagents).
- To change worktree location (e.g. `../resumind-worktrees/`), update both subagents AND the `opsx-launch-parallel` / `opsx-parallel-cleanup` commands; the planner emits paths relative to the main repo root.
- To enable auto-push or auto-PR, lift the "no push" rule in `opsx-change-implementer.md` and add `gh pr create` steps in `opsx-launch-parallel.md`.
