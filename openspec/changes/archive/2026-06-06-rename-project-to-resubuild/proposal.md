## Why

The product has been rebranded from **Resumind** to **Resubuild**, but the monorepo, package scopes, environment variables, scripts, and Supabase project identifier still carry the old name. This blocks the new brand from being the source of truth in code, leaves operators reading mixed signals in logs and package names, and makes upcoming external infra (Railway, self-hosted Supabase) inherit a stale name. Now is the right moment: the Railway + self-hosted Supabase deployment work is already in flight in `openspec/changes/deploy-railway-self-hosted-supabase`, and a fresh `project_id` is required before that change can run `supabase init` / push schema.

## What Changes

- Rename the root workspace name `resumind` Ôćĺ `resubuild` and every workspace package scope `@resumind/*` Ôćĺ `@resubuild/*` (web, api, types, schemas, resume-template, import-models, import-agent). Update every internal `workspace:*` dependency reference and every `pnpm --filter` invocation in root scripts accordingly.
- Rewrite hard-coded user-visible strings, MCP server names, log lines, and dev-server titles that say "Resumind" so the running app, MCP client config, and console output all use "Resubuild". Public product copy in the README, sample fixtures, and `.cursor/agents/README.md` follow the same name.
- Rename the local Supabase `project_id` in `supabase/config.toml` from `resumind` Ôćĺ `resubuild` and the `scripts/setup-local-env.sh` template defaults that mirror it. (**BREAKING** for developers: a fresh `supabase start` and re-seed are required, and any pre-existing local DB volume must be wiped.)
- Update environment variable names derived from the brand (`RESUMIND_*` Ôćĺ `RESUBUILD_*`) in `.env*` templates, sample fixtures, and scripts. Document the rename in `README.md` and the `monorepo-and-toolchain` capability.
- **Skip `openspec/changes/archive/`** ÔÇö historical proposals keep the old name to preserve provenance. No spec-level requirement changes; only one capability (`monorepo-and-toolchain`) gets a delta because its scenarios mention the brand.

## Capabilities

### New Capabilities

None ÔÇö this change does not introduce new product capabilities.

### Modified Capabilities

- `monorepo-and-toolchain`: brand references in workspace / package / hook scenarios (no behavior change beyond naming).
- `web-application`: brand string in the dev-server title and any user-visible copy surfaced through `apps/web` (no API or feature change).

## Impact

- `package.json` (root + every workspace) ÔÇö names and `pnpm --filter` references.
- `pnpm-lock.yaml` ÔÇö regenerated by `pnpm install` after package names change.
- `pnpm-workspace.yaml` ÔÇö no content change, but the lockfile regeneration is part of the verify loop.
- `apps/web/src/lib/api.ts`, `apps/web/src/lib/auth-session.ts`, and other files that embed the old env var names or `project_id`.
- `apps/api/src/mcp/**` ÔÇö MCP server name exposed to clients.
- `supabase/config.toml` and `scripts/setup-local-env.sh` ÔÇö local project_id and env template.
- `.env*` templates, `.samples/e2e-fixture.json`, `README.md`, `.cursor/agents/README.md`.
- All `openspec/specs/*` files that mention "Resumind" in capability prose (kept; only `monorepo-and-toolchain` and `web-application` produce delta specs).
- Local Supabase volume + seeded data ÔÇö must be wiped and re-seeded; `pnpm samples:seed` and `pnpm test:e2e` flow re-validated.
- External infra: none yet (the Railway + self-hosted Supabase change creates a new service, so it will use the new name from the start).
