## Why

The MCP server in `apps/api/src/mcp/` is implemented directly against the raw `@modelcontextprotocol/sdk`. That works, but it forces Resumind to hand-roll everything the SDK already provides as primitives: a per-session `McpServer` factory, a `StreamableHTTPServerTransport` lifecycle (`@All` route, `isInitializeRequest` branching, the `sessions` map, `onclose`/`onsessioninitialized` callbacks), a custom `register()` helper that builds `CallToolResult` envelopes, manual `ResourceTemplate` + `list` + `read` plumbing, and an `AsyncLocalStorage` shim (`mcpAuthStorage`) to bridge Nest's request scope into the SDK's callback model. None of that is unique to Resumind; the same scaffolding is duplicated by every Nest app that wants an MCP server. The result is ~600 lines of plumbing in `mcp.controller.ts` + `mcp-tools.service.ts` + `mcp-resource-handler.ts` that could be replaced by declarative `@Tool` / `@Resource` decorators on injectable services.

[`@rekog/mcp-nest`](https://github.com/rekog-labs/MCP-Nest) is the official NestJS-native wrapper around `@modelcontextprotocol/sdk`: it provides `McpModule.forRoot(...)` for bootstrapping the transport(s), `@Tool` / `@Resource` / `@Prompt` decorators with Zod-validated parameters, automatic tool discovery via Nest's DI, per-tool authorization, and an `enableJsonResponse` / `statelessMode` option that maps cleanly onto our existing `MCP_SERVER_ENABLED` env gate. The user explicitly asked for it to "reduce the boilerplate as much as possible". This change adopts it.

## What Changes

- Replace the direct dependency on `@modelcontextprotocol/sdk` with `@rekog/mcp-nest` (which pulls `@modelcontextprotocol/sdk` in as a peer dep). Keep `@modelcontextprotocol/sdk` installed only if `McpStreamableHttpService` from the wrapper requires it directly; otherwise drop it.
- Delete `apps/api/src/mcp/mcp.controller.ts`, `apps/api/src/mcp/mcp-tools.service.ts`, `apps/api/src/mcp/mcp-resource-handler.ts`, and the `resources/` subdirectory. Their responsibilities move to `@Tool` / `@Resource` providers and the wrapper's built-in controller.
- Wire `McpModule.forRoot(...)` into `apps/api/src/mcp/mcp.module.ts` (or a new `McpServerModule`) with the same transport shape we use today: Streamable HTTP on `/mcp` + `/mcp/`, stateful mode (random UUID session id) for compatibility with the existing E2E suite, `MCP_SERVER_ENABLED=false` to short-circuit boot, and the same `McpApiKeyGuard` at the application level.
- Convert each MCP tool into a dedicated `@Injectable()` provider that uses the `@Tool({ name, description, parameters })` decorator on a single method. The 20 tools in `MCP_TOOL_NAMES` map 1:1 to 20 new files under `apps/api/src/mcp/tools/` (e.g. `list-cvs.tool.ts`, `export-cv-pdf.tool.ts`). The rich descriptions stay in `tool-definitions.ts`; the wrapper's `parameters` field accepts the same Zod schemas we use today.
- Convert each MCP resource into a dedicated `@Injectable()` provider using the `@Resource({ uri, name, description, mimeType })` decorator. The three resources (`cv`, `application`, `media`) become `cv.resource.ts`, `application.resource.ts`, `media.resource.ts` under `apps/api/src/mcp/resources/`. The `list` / `read` body is unchanged; the URI template is declared on the decorator.
- Drop the `mcpAuthStorage` / `AsyncLocalStorage` shim and the `mcpAuthStorage.run(req.user, ...)` wrap in the controller. The `@Tool` / `@Resource` handlers now receive the authenticated `AuthUser` via a small `McpAuthContext` injectable that reads `req.user` from the underlying Nest request (the wrapper exposes the `Request` object on the tool context, or we use `REQUEST` injection scoped to the request). The `getMcpAuthUser()` helper stays as a thin re-export so the existing service code (`CvService`, `ApplicationService`, `MediaService`) doesn't need a single change.
- Keep the `McpApiKeyGuard` + `McpKeyRepository` path entirely intact — it's already a plain Nest guard and works the same way once the wrapper's controller is guarded by `@UseGuards(McpApiKeyGuard)` (or by a global guard attached to the module).
- Keep `McpExportService`, `ExportStorageService`, `CvExportService`, `CvTemplatePresentationService`, `ApplicationService`, `CvService`, `CvJsonResumeSwapService`, `MediaService` unchanged. The four `export_cv_*` tools continue to return the signed-URL envelope from `McpExportService.publish*`; `fetch_export_url` continues to call `McpExportService.refreshSignedUrl`. The signed-URL response shape, the per-tool `structuredContent` wrapping, and the per-tool `annotations` (`readOnlyHint` / `destructiveHint`) are preserved.
- Update unit tests: `mcp.controller.spec.ts` and `mcp-tools.service.spec.ts` are deleted (the controller goes away; the tool service is split into per-tool providers). Each new `*.tool.ts` / `*.resource.ts` ships a colocated `*.spec.ts` that covers the Zod-schema validation, the call into the underlying service, and the `structuredContent` shape. Add a single `mcp-server.module.spec.ts` that asserts all 20 tools and 3 resources are picked up by `McpModule.forRoot` discovery.
- Update `apps/api/README.md` MCP section to reference `@rekog/mcp-nest` and to drop the manual "Streamable HTTP transport" boilerplate note; the client config example (Cursor / Claude `mcp.json` snippet with `Authorization: Bearer <key>`) is unchanged.
- No DB migrations, no RLS changes, no spec-level requirement changes for MCP clients, no changes to the REST API, no changes to the web app.

## Capabilities

### New Capabilities

_None._ The change is implementation-only. The MCP tool catalog, the request/response envelopes, the resource URIs, the `McpApiKeyGuard` flow, and the signed-URL export transport all stay byte-compatible.

### Modified Capabilities

- `mcp-server`: The internal implementation of every MCP tool and resource migrates from a hand-rolled `McpServer` / `StreamableHTTPServerTransport` stack to `@rekog/mcp-nest` `@Tool` / `@Resource` providers. The public contract — tool names, Zod input schemas, response shapes (`structuredContent` + `content[]`), `annotations`, resource URIs, and the `MCP_SERVER_ENABLED` / `McpApiKeyGuard` semantics — is unchanged. The `mcp-server` delta spec captures the implementation change so future audits can see we adopted the wrapper without altering the contract.

- `apps/api/src/mcp/mcp-tools.service.ts` — deleted.

- `apps/api/src/app.module.ts` — unchanged (`McpModule` is still imported).
- `apps/api/README.md` — drop the manual transport paragraph, add a one-liner "uses `@rekog/mcp-nest` for the MCP transport" with a link to the wrapper docs.
