## Context

Release 1 of resubuild already ships a docker compose target
(`docker-compose.prod.yml`) that builds and runs the existing
`apps/api/Dockerfile` and `apps/web/Dockerfile` against a cloud
Supabase project, fed by a single `.env.prod` generated by
`pnpm setup:env:prod`. The compose target assumes a host the
operator manages: a single bridge network, a named volume for the
Puppeteer chromium cache, port-mapping from the host into the
containers, and an HTTPS terminator (out of scope for release 1)
sitting in front of the exposed ports.

Railway is a managed container platform with a different
deployment model: each service is its own deployable unit
(unrelated to `docker compose` services), the build context is
pushed by the Railway CLI (or via a GitHub integration), every
service receives an auto-generated public HTTPS domain (or a
custom domain attached manually), and the platform injects a
`PORT` env var at runtime that the process must honor. There is
no shared internal network in the docker compose sense â€” services
on Railway talk to each other over the public internet (or, on
paid plans, over a private network referenced by service name).

The two services (api and web) need to land in the same Railway
project so the operator can manage them as one unit, and the
public URLs the two services exchange (`CORS_ORIGIN`,
`APP_URL`, `PUBLIC_API_URL`, `NEXT_PUBLIC_API_URL`) must point at
Railway's public domains (or the operator's custom domains), not
at `http://localhost:*`. The env-var surface is identical to the
docker compose target â€” only the default values for the public
URLs differ.

## Goals / Non-Goals

**Goals:**

- Add a one-command deployable Railway target that reuses the
  existing `apps/api/Dockerfile` and `apps/web/Dockerfile`
  without modification.
- Centralize the Railway-specific build config in two
  `railway.json` files at each service's root directory (one
  per service, co-located with the corresponding `Dockerfile`)
  and a `.railwayignore` at the repo root that keeps the
  build context lean.
- Make the env-var contract from the docker compose target
  reusable on Railway by extending the existing
  `pnpm setup:env:prod` flow with a `--target railway` flag that
  bakes the production custom domains into the four public-URL
  keys (`CORS_ORIGIN`, `APP_URL`, `PUBLIC_API_URL`,
  `NEXT_PUBLIC_API_URL`) so the generated `.env.prod` is
  immediately deployable â€” no find-and-replace step is required
  as long as the operator attaches the matching custom domains
  to the corresponding services in the Railway dashboard.
- Ship an operator workflow (Cursor SKILL + cursor command +
  README subsection) that walks through the deploy step by step
  and reuses the existing manifest-on-disk pattern from the
  prod-env-bootstrap-helper change so secrets never round-trip
  through chat.
- Configure both services to scale to zero when idle via
  Railway's "App Sleeping" per-service dashboard toggle (not
  representable in `railway.json`). The SKILL documents the
  toggle as a per-service step the operator must enable.
- Document the known limitations of the Railway target (no
  Puppeteer cache volume on the free tier, no preview
  environments) in the spec and the SKILL â€” not as a TODO, as
  an honest scope statement.

**Non-Goals:**

- No CDN / edge caching layer in front of the web or api
  services.
- No preview / ephemeral environments per pull request. That
  belongs to a follow-on release-1 change.
- No migration of the existing docker compose target â€” it stays
  fully supported and is the recommended target for self-hosted
  operators.
- No changes to `apps/api/src` or `apps/web/src` source code.
  `process.env.PORT` is already honored by both the NestJS
  `bootstrap()` and Next.js `next start`, and the existing
  Dockerfile `EXPOSE 3000` lines remain semantically correct
  (Railway ignores `EXPOSE`; the platform pins the port via the
  injected env var).
- No Railway-specific secrets manager integration. The operator
  pastes the values from `.env.prod` into Railway's Variables tab
  per service, matching the docker compose target's
  `env_file: .env.prod` ergonomics.

## Decisions

### D1. One `railway.json` per service at the service root directory

**Decision.** Add `railway.json` at each service's root
directory (`apps/api/railway.json` and `apps/web/railway.json`).
Each file pins the build to the service-local `Dockerfile` and
pins the start command to the existing
`node apps/api/dist/main` and `pnpm --filter @resubuild/web
start` commands.

**Why.** Railway's config-as-code file follows the
service-root convention Railway's auto-detect expects ("railway.json
/ railway.toml are detected at the root of the package
directory"). Co-locating each `railway.json` with its
`Dockerfile` keeps the two files in lockstep: a rename of
`Dockerfile` only requires a matching `dockerfilePath` update
in the adjacent `railway.json`, and a developer editing one is
visually reminded of the other. Railway's per-service config
is JSON-serializable and lives in the repo (the alternative is
configuring each service entirely in the Railway dashboard,
which is non-reproducible). Railway discovers the two services
from the two `railway.json` files in the same project link.
The build step uses `builder: "DOCKERFILE"` and
`dockerfilePath: "apps/{service}/Dockerfile"` (resolved
relative to the **repo root** â€” Railway's Root Directory
service setting does not apply to paths inside
`railway.json`); the deploy step pins `startCommand` to the
same command the existing Dockerfiles declare as `CMD`.

**Alternatives considered.**

- A single multi-service `railway.json` at the repo root: not
  supported â€” Railway expects one config per service, and the two
  services in this project have different Dockerfiles, start
  commands, and env-var subsets.
- Two `railway.json` files at the repo root with names like
  `railway.api.json` / `railway.web.json` and absolute
  `dockerfilePath` values: works, but splits a per-service
  config across two non-adjacent locations and makes
  Dockerfile drift harder to spot during code review.
- A `Dockerfile.railway` per app: rejected. The existing
  `apps/api/Dockerfile` and `apps/web/Dockerfile` are
  already a clean fit (multi-stage, Puppeteer headless shell,
  production `pnpm install --prod`); a parallel Dockerfile would
  drift. The Railway target shares both Dockerfiles verbatim.
- A `railway.toml`: Railway accepts both `.json` and `.toml`;
  JSON is the documented default and parses without extra tooling.

### D2. `.railwayignore` mirrors `.dockerignore` patterns

**Decision.** Add a `.railwayignore` at the repo root that
excludes `node_modules`, `.turbo`, `.next`, `.git`, `coverage`,
`**/dist`, and the existing `node_modules/.cache` directories.

**Why.** The build context for both services is the repo root
(both Dockerfiles expect a monorepo context with `packages/` and
`apps/` available). Without an ignore file, the Railway CLI
uploads the entire monorepo on every push, which is slow and
unnecessary â€” the multi-stage Dockerfiles rebuild what's needed
inside the container.

**Alternatives considered.**

- Reusing `.gitignore`: Railway reads `.gitignore` as a fallback
  for some files but is stricter than the docker daemon about
  context exclusions. A dedicated `.railwayignore` makes the
  intent explicit and survives a future change that adds
  build-time files to git (e.g. lockfile-only commits).
- Configuring the build to use a subdirectory context: rejected.
  Both Dockerfiles `COPY packages ./packages` and
  `COPY apps ./apps` from the monorepo root, so the build context
  _must_ be the repo root.

### D3. `--target railway` flag on the existing env generator

**Decision.** Extend `scripts/setup-prod-env.mjs` and
`scripts/lib/env-prod-schema.mjs` with a `--target <name>` flag
that defaults to `docker-compose` (the existing behavior) and
accepts `railway` as a second value. When the operator passes
`--target railway`, the generator writes the **production
custom domains** as the default values for `CORS_ORIGIN`,
`APP_URL`, `PUBLIC_API_URL`, and `NEXT_PUBLIC_API_URL`:

- `CORS_ORIGIN` and `APP_URL` â†’ `https://app.resubuild.dev` (the
  web app's production custom domain)
- `PUBLIC_API_URL` and `NEXT_PUBLIC_API_URL` â†’
  `https://api.resubuild.dev` (the API's production custom
  domain)

The manifest always wins â€” if the operator supplies any of these
four keys in `prod-secrets.json`, those values are used verbatim.
The generated `.env.prod` is immediately deployable as long as
the operator has attached `app.resubuild.dev` to the web service
and `api.resubuild.dev` to the api service in the Railway
dashboard (Step 4c of the SKILL). No find-and-replace step is
required.

**Why.** The env-var surface is identical between the docker
compose target and the Railway target â€” only the default values
for the public URLs differ. Reusing the existing generator keeps
a single source of truth in
`scripts/lib/env-prod-schema.mjs` and avoids a parallel
`setup-railway-env.mjs` that would drift. The two custom domains
are stable, known production endpoints; baking them into the
generator's Railway target eliminates the
"find-and-replace the placeholders after the first deploy" step
that would otherwise be required, and makes the generated
`.env.prod` directly usable by both the local docker compose
target and the Railway target with no per-target edit.

**Alternatives considered.**

- A separate `setup:env:railway` script: rejected. The manifest
  schema would be duplicated, and the SKILL / cursor command
  surface would multiply.
- Keep `*.up.railway.app` placeholders and require a
  find-and-replace step: rejected. The two production custom
  domains are already known and stable; the placeholder round
  trip adds operator friction for no real benefit (the operator
  still has to know the Railway-printed domain after the first
  deploy to do the find-and-replace, which the SKILL cannot
  automate).
- Embed the two custom-domain values as a per-deployment
  override on the `pnpm deploy:railway` script: rejected. The
  generator is the right place for default values that are
  stable across deploys; the preflight script is a guard, not a
  source of defaults.

### D4. Operator workflow reuses the manifest-on-disk pattern

**Decision.** The new
`.cursor/skills/railway-deploy/SKILL.md` and
`.cursor/commands/railway-deploy.md` reuse the
`prod-secrets.json` pattern from
`.cursor/skills/setup-prod-env/SKILL.md` so the same manifest
feeds both the docker compose target and the Railway target.

**Why.** The manifest is the only place secret material lives
during the deploy flow, the file is gitignored, and the SKILL
never echoes secret values back to the chat. The Railway SKILL
just adds three steps on top of the existing flow: create the
Railway project, create the two services, paste the manifest
values into the per-service Variables tab.

**Alternatives considered.**

- A separate `railway-secrets.json`: rejected. Operators would
  have to maintain two manifests for the same env-var surface.
  The existing `prod-secrets.json` is target-agnostic and the
  generator's `--target` flag chooses the URL defaults.

### D5. `pnpm deploy:railway` is a thin wrapper, not a deploy

**Decision.** Add a `deploy:railway` script to the root
`package.json` that: (1) checks `.env.prod` exists, (2) prints
the next two `railway up --service <name>` commands, (3) exits.
The script does not call `railway up` itself.

**Why.** The Railway CLI's deploy command is interactive and
service-scoped. Wrapping it in a non-interactive script would
either (a) hide failures or (b) require a new flag set the
operator has to learn. The thin wrapper just validates the
preflight and prints the next command â€” the operator runs the
deploy interactively and watches the build log.

**Alternatives considered.**

- A fully non-interactive `pnpm deploy:railway` that calls
  `railway up` for both services: rejected. The Railway CLI's
  auth model (project link, service link, environment selection)
  is interactive by design; trying to drive it non-interactively
  adds complexity for no operator benefit.
- No wrapper script at all: rejected. The preflight check
  (`.env.prod` exists, contains the required keys) is a useful
  guard before the operator starts a 2â€“3 minute Railway build.

### D6. App Sleeping (scale to zero) is a per-service dashboard toggle, not a `railway.json` field

**Decision.** Document the App Sleeping per-service dashboard
toggle in the SKILL (Step 4d) and the README's Railway
subsection, but do not represent it in `railway.json` or any
other config-as-code file.

**Why.** Railway's config-as-code schema (`railway.json` /
`railway.toml`) does not include a scale-to-zero / sleep field.
The supported settings are `build`, `deploy.startCommand`,
`deploy.healthcheckPath`, `deploy.multiRegionConfig`, and
`deploy.deploymentTeardown`. The scale-to-zero behavior is a
per-service property ("App Sleeping") configured in the
dashboard at **Settings â†’ App Sleeping** that goes into effect
on the next deploy. Trying to encode it in `railway.json`
would either be silently ignored (if we hand-rolled a non-
Railway field) or be a maintenance liability (if Railway ever
adds a real field, our hand-rolled key would conflict).

**Alternatives considered.**

- A custom `sleep: true` field in `railway.json` ignored by
  Railway but checked by our preflight script: rejected. The
  preflight script does not have the authority to call the
  Railway API to actually flip the toggle, so the check would
  be advisory only and could give false confidence.
- Forcing the service to always-on and skipping scale-to-zero
  for release 1: rejected. App Sleeping is the natural fit for
  a small-team release-1 target that does not run 24/7; the
  alternative would mean a meaningful monthly cost on the
  paid plan or a degraded UX on the free tier (no persistent
  volume either way).
- A `railway variables set` / `railway api` step in
  `pnpm deploy:railway` that toggles App Sleeping: rejected.
  The Railway API for App Sleeping is not currently part of
  the stable public API; relying on it would couple the
  release-1 SKILL to an unstable surface.

## Risks / Trade-offs

- **No Puppeteer cache volume on Railway free tier** â†’
  Mitigation: the SKILL and the spec document that the api
  service will re-download Chromium on every cold start from
  App Sleeping (~181 MiB per the Dockerfile comment, plus the
  Chromium dependency tree). The operator can opt into a paid
  Railway plan with a persistent volume mounted at
  `/root/.cache/puppeteer`, or accept the cold-start cost on
  the free tier. The cold-start penalty is documented as an
  expected behavior, not a bug.
- **No shared internal network on Railway free tier** â†’
  Mitigation: the api and web services communicate over the
  public Railway domain, which means `CORS_ORIGIN` /
  `APP_URL` / `NEXT_PUBLIC_API_URL` must point at the public
  domain (not `http://api.railway.internal` or similar). The
  generator's `--target railway` flag bakes the production
  custom domains into these four keys.
- **App Sleeping must be enabled per service by the operator
  (no `railway.json` field encodes it)** â†’ Mitigation: the
  SKILL (Step 4d) and the README's Railway subsection both
  document the toggle explicitly with the dashboard path. The
  `pnpm deploy:railway` preflight does not (and cannot)
  verify the toggle is on; the operator is responsible.
- **Custom-domain attachment is a one-time per-service
  dashboard step** â†’ Mitigation: the SKILL (Step 4c) lists the
  two `app.resubuild.dev` and `api.resubuild.dev` attachments
  as required. The generator writes these exact domains as
  defaults, so a missing attachment is the only thing that can
  break the deploy â€” and the operator can see it immediately
  in the `curl` verification step.
- **Two `railway.json` files to keep in sync** â†’ Mitigation: the
  files are small (one builder config, one start command, one
  healthcheck path) and the unit test in
  `scripts/lib/railway-config.spec.mjs` (task 1.3) validates the
  shape on every CI run.
- **Operator confusion about which target is which** â†’ Mitigation:
  the SKILL and the README subsection explicitly call out that
  `--target railway` is for the Railway target and that the
  default (`--target docker-compose`, implicit) is for the
  docker compose target. The two `.env.prod` files are
  interchangeable â€” only the default URL values differ, and
  those are now real production custom domains for the
  Railway target.
- **`.railway/` directory created by the CLI** â†’ Mitigation: the
  directory is added to `.gitignore` next to the existing
  `.env.prod` and `prod-secrets.json` entries.

## Migration Plan

This is a new deployment target â€” there is no migration from an
existing target. The deploy steps an operator follows:

1. Run `pnpm setup:env:prod --target railway --from
prod-secrets.json` to generate `.env.prod` with the
   production custom domains baked in for the four public-URL
   keys.
2. Install the Railway CLI: `npm install -g @railway/cli` (or
   `brew install railway`).
3. `railway login` (interactive, opens browser).
4. Use the Railway dashboard to create a project (the SKILL
   walks through the dashboard path because `railway init`
   does not handle the "one project, two services from the
   same repo" pattern cleanly).
5. `railway link` from the repo root, then create the `api`
   service: `railway add --service api`. Point the service at
   `apps/api/railway.json` (Railway's config-as-code
   auto-detect locates this file at the service root
   directory; the `dockerfilePath` it contains is resolved
   from the repo root, so the value is the repo-root-relative
   `apps/api/Dockerfile`).
6. `railway add --service web`. Point the service at
   `apps/web/railway.json` (the `dockerfilePath` is the
   repo-root-relative `apps/web/Dockerfile`).
7. **Attach the custom domains**: in the api service's
   **Settings â†’ Networking â†’ Custom Domain**, attach
   `api.resubuild.dev` (Railway provisions a Let's Encrypt
   certificate automatically). Repeat for the web service with
   `app.resubuild.dev`.
8. **Enable App Sleeping**: in each service's
   **Settings â†’ App Sleeping**, toggle **Enable App Sleeping**
   on. Trigger a new deployment for the change to take
   effect.
9. For each service, open the Variables tab and paste the
   corresponding subset of `.env.prod` values (the SKILL lists
   which keys go where). The SKILL also includes a
   `railway variables set KEY=VALUE` loop for operators who
   prefer the CLI.
10. Run `pnpm deploy:railway` to validate the preflight, then
    `railway up --service api` and `railway up --service web`
    (the SKILL prints both commands).
11. Verify with `curl https://api.resubuild.dev/_health` and
    `curl https://app.resubuild.dev/`.

**Rollback.** The Railway target is additive â€” rolling back means
deleting the two services in the Railway dashboard. The docker
compose target is unaffected.

## Open Questions

- Should the `railway.json` files pin a specific Railway
  environment (`production` vs `staging`)? The default Railway
  behavior is to deploy to a `production` environment; the
  `railway.json` does not need to declare it. If the operator
  later adds a `staging` environment, the deploy:railway
  script would need a `--environment` flag. Deferred to a
  follow-on change.
- Should the api service's Puppeteer cache live in a Railway
  volume (paid feature) or in a S3-compatible bucket? Out of
  scope for release 1 â€” the current code does not implement a
  cache backing store. The existing in-container cache works
  for the docker compose target; on Railway it just regenerates
  per deploy.
- Does the operator want a one-click "Deploy to Railway" button
  in the README? Deferred â€” that requires a Railway template,
  and Railway templates do not cleanly support a single-repo
  two-service deploy. The SKILL is the practical equivalent.
