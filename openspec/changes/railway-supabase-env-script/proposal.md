## Why

A new `railway-supabase-env-script` script must exist to generate the env vars a developer needs to paste into the Railway service variables panel when running a self-hosted Supabase on Railway. Today, generating these values requires running `supabase start` locally, copying values out of `supabase status -o env`, and then hand-deriving JWT secrets, service-role keys, and per-service URLs — a flow that does not work for a fresh Railway project where the only available Supabase is the self-hosted one being deployed. The user wants a single command that prints (or writes) the canonical env-var set: per-service `SITE_URL`, `API_EXTERNAL_URL`, `SUPABASE_PUBLIC_URL`, `POSTGRES_PASSWORD`, `JWT_SECRET`, `ANON_KEY`, `SERVICE_ROLE_KEY`, `SECRET_KEY_BASE`, `PG_META_CRYPTO_KEY`, plus the API-side `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_JWT_SECRET`, `AI_AGENT_ENCRYPTION_KEY`, `MEDIA_BUCKET`, `MCP_EXPORT_BUCKET`, `CORS_ORIGIN`, `APP_URL`, `PUBLIC_API_URL` so a Railway project is one paste away from a running stack.

## What Changes

- Add `scripts/generate-railway-supabase-env.mjs` — a Node script that prints (or writes to `.env.railway`) the env vars a developer must set on each Railway service. It generates the secrets that are normally generated by `supabase start` / `supabase init` (Postgres password, JWT secret, anon key, service-role key, secret-key-base, pg-meta crypto key) and the API-side keys that are normally auto-filled by `scripts/setup-local-env.sh` (Supabase URL/keys, AI agent encryption key, bucket names, public URLs).
- The script is **offline-capable**: it generates keys itself (via `node:crypto` random bytes) so the developer can run it before any Supabase service is up. Generated JWTs are HS256-signed with the script-generated `SUPABASE_JWT_SECRET` so they match what GoTrue will validate against once the auth service starts.
- Add a `pnpm railway:env` npm script at the repo root that runs the generator. The script supports a `--output <path>` flag (default stdout) and a `--format dotenv|shell|json` flag (default `dotenv` so it can be pasted straight into the Railway "RAW Editor" or copied into a `.env.railway` file).
- Add `scripts/README.md` (or extend an existing scripts doc) with a one-section explanation of how to run `pnpm railway:env` and which Railway services each block of env vars belongs to (`web`, `api`, `auth` (GoTrue), `rest` (PostgREST), `storage`, `realtime`, `db`, `studio`, `imgproxy`, `kong`, `supabase-admin`).
- **No application-level code changes**: the Nest API, Next.js web, Supabase migrations, RLS policies, REST contracts, MCP server, and import agent stay byte-identical to local development. The script only generates environment variables.
- **No new npm dependencies**: the script uses `node:crypto` and `node:fs` only.
- **BREAKING (no)**: this is a pure additive developer-experience change. It does not change runtime behavior of the API, web, or Supabase services.

## Capabilities

### New Capabilities

- `railway-supabase-env`: A documented, reproducible way to generate the env vars a developer must paste into Railway's per-service variable panel when deploying self-hosted Supabase + Resumind (Next.js + Nest API) to Railway. Covers the secrets the Supabase self-hosting stack expects (Postgres, GoTrue, PostgREST, Storage, Realtime, Studio, imgproxy, Kong, supabase-admin) and the env vars the Resumind API and web app read (`SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_JWT_SECRET`, `AI_AGENT_ENCRYPTION_KEY`, `MEDIA_BUCKET`, `MCP_EXPORT_BUCKET`, `CORS_ORIGIN`, `APP_URL`, `PUBLIC_API_URL`, `NEXT_PUBLIC_*`).

### Modified Capabilities

- `monorepo-and-toolchain`: Add a new root script `pnpm railway:env` that runs `scripts/generate-railway-supabase-env.mjs`. The script is **not** part of `pnpm verify` and is **not** run in CI; the verify pipeline and CI jobs (format, lint, typecheck, test, build) stay unchanged.

## Impact

- **New files**:
  - `scripts/generate-railway-supabase-env.mjs` — Node ESM script. Generates the env-var set, prints to stdout or writes a file. Pure `node:crypto` + `node:fs`; no npm deps.
  - `scripts/lib/railway-env.mjs` — shared helpers (HS256 JWT sign using `crypto.createHmac('sha256', ...).update(`${header}.${payload}`).digest('base64url')`, base64url encoding, random-key generators, dotenv/shell/json formatters, default config for service hostnames on Railway).
  - `scripts/generate-railway-supabase-env.mjs` is a thin wrapper that imports `./lib/railway-env.mjs`, generates the canonical env-var map, and emits it in the requested format.
- **Modified files**:
  - `package.json` (root) — add `"railway:env": "node scripts/generate-railway-supabase-env.mjs"` to `scripts`.
  - `scripts/README.md` (new) or the existing `scripts/setup-local-env.sh` comment block — short "Railway env generation" section pointing at `pnpm railway:env`.
- **No backend domain logic changes**: `apps/api/src/` modules (auth, CV, media, MCP, import agent, export) are untouched. The API's `ConfigModule` already reads the same env-var names from `apps/api/.env` or from the Railway env-var panel — no change to how config is loaded.
- **No web app feature changes**: `apps/web/src/` pages, components, hooks, and TanStack Query helpers are untouched. The web's `NEXT_PUBLIC_*` env vars appear in the printed env block so the developer can paste them into the Railway `web` service, but no new env var is read.
- **No database schema changes**: `supabase/migrations/` stays as-is. The generated `POSTGRES_PASSWORD` and `JWT_SECRET` are written to the `db` and `auth` (GoTrue) Railway service env vars; the migrations themselves are pushed later by the bootstrap step (handled by a separate change, not in scope here).
- **No spec-level behavior changes** for end users other than: a developer can now run one command to get the full env-var set for a Railway deployment, instead of hand-deriving secrets.
- **Security**: the generated secrets are sensitive and the script prints them to stdout (or writes them to a file in `.gitignore`). The script is documented as: do not commit the output; rotate the keys if they leak; Railway env vars are the secure store of record.
- **Out of scope**: pushing the migrations, creating the `media` / `mcp-exports` Storage buckets, seeding the developer / E2E accounts, or starting any Railway service. Those are handled by the separate `deploy-railway-self-hosted-supabase` change / bootstrap script. This change is **only** the env-var generation step.
